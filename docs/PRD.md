# PainRadar 产品需求文档

## 1. 产品概述

PainRadar 是一款面向创业者、产品经理、独立开发者和行业研究者的痛点捕获与机会洞察工具。产品将用户日常遇到的抱怨、摩擦和不便沉淀为结构化痛点库，并通过行业趋势、周期性模式和 AI 洞察帮助用户识别潜在市场机会。

当前前端版本以移动端体验为优先，围绕三个核心场景构建：

- 捕获：快速记录一个真实痛点或抱怨。
- 沉淀：在痛点库中浏览、分类、查看详情。
- 洞察：聚合痛点数据，发现行业趋势、摩擦模式和创业机会草案。

## 2. 产品目标

### 2.1 业务目标

- 降低痛点记录门槛，让用户能在 10 秒内完成一次痛点捕获。
- 将零散抱怨转化为可浏览、可分析、可追踪的机会资产。
- 通过 AI 洞察和机会评分强化产品差异化，提升用户复访动力。
- 为后续接入真实数据库、认证体系、AI 分析和协作功能预留清晰前端边界。

### 2.2 用户目标

- 随时记录生活或工作中观察到的真实问题。
- 快速回看自己的痛点库，按行业、时间、影响力理解痛点分布。
- 查看单个痛点背后的标签、位置、关注度和商业洞察。
- 从聚合趋势中发现值得进一步验证的创业方向。

## 3. 目标用户

### 3.1 创业者

需要从真实用户抱怨中寻找市场缺口、验证问题强度，并形成早期方案草案。

### 3.2 产品经理与 UX 研究者

需要长期收集用户摩擦点，并把分散反馈整理为可分析的需求线索。

### 3.3 独立开发者

需要轻量发现可做的小产品机会，关注高频、具体、可解决的问题。

### 3.4 行业研究者

需要观察行业级痛点热度、增长趋势和周期性摩擦模式。

## 4. 范围说明

### 4.1 当前前端范围

- 痛点捕获页。
- 痛点库列表页。
- 痛点详情底部弹层。
- 洞察仪表页。
- 底部主导航。
- 深色主题、雷达视觉语言、玻璃拟态卡片风格。
- 登录/注册页面和受保护路由基础能力保留。

### 4.2 当前不在范围内

- 真实痛点持久化保存。
- 搜索、筛选、排序的完整交互实现。
- 分享、收藏、生成方案草案的真实后端逻辑。
- 多人协作、团队空间和权限管理。
- 真实地图定位与地理聚合分析。
- 真实 AI 模型调用和机会评分算法。
- Todo 模板页面作为产品功能对外暴露。

## 5. 信息架构

### 5.1 主导航

底部导航包含三个一级入口：

- 捕获：`/`
- 库：`/library`
- 洞察：`/insights`

### 5.2 辅助路由

- 痛点详情：`/library/$id`
- 登录注册：`/login`
- 受保护仪表页：`/_auth/dashboard`

## 6. 核心页面需求

### 6.1 痛点捕获页

路径：`/`

用户价值：让用户以最低成本记录刚刚观察到的痛点。

功能需求：

- 展示品牌主张：“每一个抱怨，都是一个尚未被满足的市场机会。”
- 提供多行文本输入框，用于记录痛点描述。
- 输入为空时点击保存，应提示用户先填写内容。
- 输入有效时点击保存，应清空输入框并提示保存成功。
- 提供进入痛点库的文字入口。
- 页面底部展示主导航，并高亮“捕获”。

交互要求：

- 保存按钮必须有明确点击反馈。
- 输入区域应适配移动端单手输入。
- 空内容校验应即时、轻量，不打断用户流程。

验收标准：

- 用户打开首页即可看到输入框和保存按钮。
- 空输入点击保存显示提示：“先写下一个让你烦恼的问题吧”。
- 有效输入点击保存后输入框清空，并显示：“已保存到你的痛点库”。
- 点击“查看我的痛点库”进入 `/library`。

### 6.2 痛点库列表页

路径：`/library`

用户价值：帮助用户回看已记录的痛点，并从历史记录中发现模式。

功能需求：

- 顶部展示页面标题、搜索入口和账户入口。
- 展示统计概览：总记录、高影响、周环比。
- 展示排序入口，默认文案为“按时间排序”。
- 提供新增痛点入口，点击返回捕获页。
- 以卡片列表展示痛点记录，包括：
  - 分类标签。
  - 记录时间。
  - 痛点摘要。
  - 底部状态信息，例如发生次数、高摩擦感、洞察就绪。
  - 详情入口。
- 在列表中展示自动聚类卡片，用于提示周期性模式。
- 页面底部展示主导航，并高亮“库”。

交互要求：

- 点击痛点卡片详情入口打开详情底部弹层。
- 列表布局需要兼容移动端滚动浏览。
- 搜索、排序入口当前可作为视觉入口保留，后续接入真实交互。

验收标准：

- 页面至少展示一组统计数据和多条痛点记录。
- 点击新增按钮进入 `/`。
- 点击记录详情入口进入 `/library/$id`。
- 自动聚类卡片能与普通痛点卡片区分。

### 6.3 痛点详情底部弹层

路径：`/library/$id`

用户价值：让用户理解单个痛点的背景、商业启发和潜在关注度。

功能需求：

- 以底部弹层形式覆盖在痛点库上方。
- 展示关闭、分享、收藏操作入口。
- 展示主标签、检测时间、详情标题和详情正文。
- 展示关联话题标签。
- 展示“雷达洞察”模块，包括洞察文本和关注者数量。
- 展示位置地图视觉模块和位置信息。
- 底部固定展示“生成方案草案”主按钮。
- 当 ID 不存在时，应进入路由未找到状态。

交互要求：

- 点击关闭按钮返回 `/library`。
- 分享、收藏、生成方案草案当前可保留为提示型交互。
- 弹层高度应适配移动端，内容区域可滚动，主按钮固定在底部。

验收标准：

- 访问有效详情路径时展示对应痛点数据。
- 点击关闭按钮后回到痛点库列表。
- 点击“生成方案草案”显示“方案草案生成即将上线”提示。

### 6.4 洞察页

路径：`/insights`

用户价值：帮助用户从个人和聚合痛点数据中发现更高层次的市场机会。

功能需求：

- 展示 PainRadar 品牌头部和账户入口。
- 展示雷达扫描概览卡片，包括市场缺口、高潜力需求、趋势同步等信息。
- 展示 AI 智能洞察卡片，突出推荐关注领域和关键理由。
- 展示热门痛点行业列表，包括行业名称、痛点标记数量、增长比例和图标。
- 展示周期性摩擦模式列表，包括模式描述和强度进度条。
- 展示高潜力创业机会草案列表，包括机会名称、描述、评分和标识图。
- 提供“深入探索该领域”和“查看全部”入口。
- 页面底部展示主导航，并高亮“洞察”。

交互要求：

- 点击“深入探索该领域”显示“领域探索即将上线”提示。
- 点击“查看全部”显示“完整列表即将上线”提示。
- 页面应保持仪表盘式信息密度，优先支持快速扫描。

验收标准：

- 页面至少包含概览、AI 洞察、行业、摩擦模式、机会草案五类内容。
- 所有列表项有稳定唯一 key。
- 主导航能在捕获、库、洞察之间切换。

## 7. 数据模型

### 7.1 痛点记录

前端当前使用静态数据结构 `PainPoint`：

```ts
interface PainPoint {
  id: string;
  tags: {
    label: string;
    tone: "primary" | "secondary" | "tertiary" | "muted";
  }[];
  time: string;
  summary: string;
  detailTitle: string;
  detailBody: string;
  hashtags: string[];
  insight: string;
  watchers: number;
  location: string;
  footer: {
    icon: "history" | "priority" | "lightbulb" | "location";
    label: string;
  };
}
```

### 7.2 后续建议字段

- `createdAt`：真实创建时间。
- `updatedAt`：更新时间。
- `source`：痛点来源，例如手动记录、导入、访谈、社媒。
- `severityScore`：痛点强度。
- `frequencyScore`：出现频率。
- `opportunityScore`：机会评分。
- `status`：未分析、已聚类、已生成洞察、已生成方案。
- `ownerId`：用户 ID。

## 8. 视觉与体验要求

- 移动端优先，最大内容宽度控制在适合手机和窄屏平板的范围。
- 默认深色主题，强化雷达、扫描、洞察、机会发现的产品心智。
- 使用底部导航承载核心路径，减少移动端顶部导航负担。
- 主要操作按钮使用清晰图标和高对比度色彩。
- 卡片信息层级应清楚：标签、标题、正文、状态、行动入口不可混淆。
- Toast 反馈用于轻量提示，不替代关键状态展示。
- 交互控件必须有可访问名称，例如图标按钮使用 `aria-label`。

## 9. 非功能需求

### 9.1 性能

- 首屏应优先渲染核心输入、列表或洞察内容。
- 图片资源需要设置尺寸或稳定容器，避免布局跳动。
- 后续接入远程数据时，应为列表和详情提供加载态与错误态。

### 9.2 可访问性

- 图标按钮必须有语义标签。
- 表单控件需要明确用途和可聚焦状态。
- 文本与背景需要保持足够对比度。
- 底部弹层应支持键盘关闭和焦点管理，后续需要补齐。

### 9.3 可维护性

- 页面应继续沿用 TanStack Router 文件路由。
- UI primitives 应优先复用 `packages/ui` 中的共享组件。
- 静态 mock 数据可逐步迁移为 tRPC 查询结果。
- 新增功能应避免和模板 `todos` 功能耦合。

## 10. 后续迭代建议

### 10.1 V1 数据闭环

- 将捕获页保存行为接入后端和数据库。
- 痛点库改为读取当前用户真实记录。
- 详情页支持收藏、分享和删除。
- 增加搜索、行业筛选、时间排序。

### 10.2 V2 AI 洞察

- 自动提取标签、行业、情绪和严重程度。
- 支持相似痛点聚类。
- 生成机会摘要、目标用户、竞品假设和 MVP 建议。
- 洞察页改为基于真实数据生成趋势。

### 10.3 V3 协作与验证

- 支持团队空间。
- 支持邀请成员共同维护痛点库。
- 支持将痛点转为验证实验。
- 支持导出研究报告或机会看板。

## 11. 成功指标

- 首次记录完成率。
- 人均痛点记录数。
- 痛点详情打开率。
- 洞察页访问率。
- 生成方案草案点击率。
- 7 日留存率。
- 搜索、筛选、收藏等高意图行为使用率。

## 12. 风险与依赖

- 当前多处功能为静态 mock 或提示型交互，需要后端和 AI 能力承接。
- 外链图片资源存在不可控风险，后续建议迁移到稳定资产服务。
- 认证体系已有基础代码，但主产品流程尚未强制登录，需要明确匿名记录与登录记录的产品策略。
- Todo 模板页面仍存在于路由中，正式发布前应隐藏、移除或改造为产品内功能。
- 地图、趋势、评分当前为视觉表达，需定义真实算法和数据来源后才能作为决策依据。
